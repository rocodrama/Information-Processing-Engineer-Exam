ROW_NUMBER()는 OVER() 안의 ORDER BY 기준으로 정렬한 뒤 중복 없이 1부터 순번을 매긴다. 동점이 있어도 무조건 서로 다른 번호가 부여된다는 점이 RANK/DENSE_RANK와 다르다. 바깥쪽 ORDER BY RN이 없으면 출력 행 순서가 보장되지 않으므로 실전에서는 반드시 명시한다.

인덱스는 "조회는 빠르게, 대신 쓰기(INSERT/UPDATE/DELETE)는 느리게" 만드는 트레이드오프다. 값의 종류가 몇 개 안 되는 컬럼(성별 등)은 인덱스를 걸어도 결과 행이 테이블의 상당수를 차지해 옵티마이저가 오히려 풀스캔을 선택하는 경우가 많다.
기출변형 옵티마이저가 인덱스를 두고도 테이블 풀스캔(Full Scan)을 선택하는 경우를 설명하시오.

정답
조회 조건을 만족하는 행의 비율(선택도)이 높을 때, 또는 테이블 자체가 작을 때 옵티마이저는 인덱스 스캔 대신 풀스캔을 선택한다.
인덱스 스캔은 "인덱스에서 위치를 찾은 뒤 실제 데이터 블록을 다시 읽는" 추가 비용(랜덤 I/O)이 든다. 조건에 맞는 행이 테이블의 대부분을 차지한다면, 이 추가 비용이 누적되어 오히려 처음부터 순서대로 쭉 읽는 풀스캔(순차 I/O)보다 비효율적이다. 그래서 옵티마이저는 통계 정보를 바탕으로 예상 비용이 더 낮은 방식을 자동으로 선택한다.

맨 앞의 LEFT JOIN은 "주문 없는 고객도 보존"하려는 의도였지만, 바로 뒤에 이어진 일반 JOIN(=INNER JOIN)이 결제 기록이 없는 행을 전부 걸러내며 그 의도를 무력화시킨다. 취소된 주문(이, O2)은 결제 자체가 없어 탈락하고, 주문이 아예 없는 박(NULL)은 o.OID = pay.OID가 NULL = pay.OID가 되어 항상 UNKNOWN 처리되어 역시 탈락한다. 결과적으로 "모든 고객을 보여주고 싶었지만" 실제로는 결제까지 완료한 고객만 남는 대표적인 LEFT JOIN 무력화 함정이다. 의도대로 만들려면 두 번째 조인도 LEFT JOIN으로 바꿔야 한다.
