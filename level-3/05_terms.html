01 요구사항 확인
기출 "액터(Actor)와 시스템 간의 상호작용을 통해 시스템이 제공해야 할 기능적 요구사항을 표현하는 UML 행위 다이어그램"
▼
정답
유스케이스 다이어그램 (Use Case Diagram)
시스템 외부의 사용자나 외부 시스템(액터)이 시스템과 어떤 방식으로 상호작용하는지를 시각화하여 기능 요구사항을 도출하는 데 사용됩니다.
기출 "클래스의 속성, 오퍼레이션(메서드)과 클래스 간의 관계(연관, 상속 등)를 표현하는 UML 구조 다이어그램"
▼
정답
클래스 다이어그램 (Class Diagram)
시스템의 정적인 구조를 표현하는 대표적인 구조(Structural) 다이어그램으로, 객체지향 설계의 기본 골격이 됩니다.
기출 "객체들 사이에 주고받는 메시지를 시간의 흐름 순서에 따라 표현하는 UML 행위 다이어그램"
▼
정답
시퀀스 다이어그램 (Sequence Diagram)
세로축은 시간의 흐름, 가로축은 참여하는 객체를 나타내며 객체 간 메시지 교환 순서를 표현합니다. 상태 변화를 보는 상태 다이어그램과 혼동하지 않도록 주의합니다.
기출 "전체와 부분의 관계이지만, 전체 객체가 소멸되어도 부분 객체는 독립적으로 존재할 수 있는 클래스 다이어그램 관계"
▼
정답
집합 관계 (Aggregation)
속이 빈 마름모(◇)로 표기합니다. 전체가 사라지면 부분도 함께 사라지는 강한 결합의 복합(Composition, 채운 마름모 ◆) 관계와 구분해야 합니다.
기출 "전체 객체가 소멸되면 부분 객체도 함께 소멸되는 강한 전체-부분 관계를 나타내는 클래스 다이어그램 관계"
▼
정답
복합 관계 (Composition)
채운 마름모(◆)로 표기하며, 집합 관계보다 강한 소유 관계입니다. 예: 집(전체)이 없어지면 방(부분)도 존재할 수 없는 관계.
기출 "진행자(Moderator)가 주도하며 정해진 체크리스트를 기반으로 결함을 찾아내는, 요구사항 검증 기법 중 가장 공식적인 방법"
▼
정답
인스펙션 (Inspection)
작성자와 검토자의 역할이 분리된 공식적 검토 기법입니다. 공식성 순서는 동료검토 < 워크쓰루 < 인스펙션 순으로 강해집니다.
기출 "시스템이 '무엇을 해야 하는가'가 아니라 응답시간, 가용성, 보안 등 '얼마나 잘 수행해야 하는가'를 정의하는 요구사항"
▼
정답
비기능적 요구사항 (Non-Functional Requirement)
기능적 요구사항이 시스템의 동작·처리 내용을 정의하는 것과 대비되며, 성능·보안·가용성·유지보수성·이식성 등이 대표적인 세부 항목입니다.
기출 "유스케이스 다이어그램에서 시스템과 상호작용하는 외부의 사람이나 다른 시스템"
▼
정답
액터 (Actor)
사람뿐 아니라 외부의 다른 시스템, 시간(스케줄러) 등도 액터가 될 수 있습니다.
심화 "기본 유스케이스가 실행될 때 다른 유스케이스의 기능을 반드시 포함하여 수행하는 유스케이스 관계"
▼
정답
include 관계 (포함 관계)
점선 화살표에 <<include>>로 표기하며 항상 실행됩니다. 특정 조건에서만 선택적으로 실행되는 <<extend>>(확장) 관계와 구분해야 합니다.
심화 "소프트웨어 요구사항 명세서(SRS)의 특성 중, 각 요구사항의 출처를 파악하고 이후 설계·구현 단계까지 연결하여 확인할 수 있는 성질"
▼
정답
추적 가능성 (Traceability)
요구사항 변경이 설계·구현·테스트에 미치는 영향을 파악할 수 있게 해줍니다. 정확성·완전성·일관성·검증 가능성 등과 함께 SRS의 주요 품질 특성입니다.
02 데이터 입출력 구현
기출 "테이블의 모든 속성값이 더 이상 분해할 수 없는 원자값(Atomic Value)을 갖도록 반복 그룹을 제거하는 정규화 단계"
▼
정답
제1정규형 (1NF)
한 컬럼에 여러 값(다중값)이 들어가는 반복 그룹을 제거하는 것이 핵심입니다.
기출 "기본키(복합키)의 일부에만 종속되는 부분 함수 종속을 제거하여 완전 함수 종속만 남기는 정규화 단계"
▼
정답
제2정규형 (2NF)
복합키의 일부에만 종속된 속성을 별도 테이블로 분리합니다. 예: (학번, 과목코드) → 성적은 완전 함수 종속이지만, 학번 → 학생명은 부분 함수 종속이라 분리 대상입니다.
기출 "A→B, B→C가 성립할 때 A→C가 되는 이행 함수 종속을 제거하는 정규화 단계"
▼
정답
제3정규형 (3NF)
기본키가 아닌 속성이 다른 기본키 아닌 속성을 결정하는 이행 종속 관계를 분리합니다.
심화 "제3정규형을 만족하면서, 테이블 내의 모든 결정자가 후보키가 되도록 만드는 정규형"
▼
정답
BCNF (보이스-코드 정규형, Boyce-Codd Normal Form)
후보키가 아닌 속성이 결정자로 작용하는 경우를 분리합니다. 3NF보다 조금 더 엄격한 조건입니다.
기출 "정규화된 테이블에 중복 컬럼이나 파생(계산) 컬럼을 추가하여 조회 성능을 높이지만, 데이터 무결성이 저하될 수 있는 기법"
▼
정답
반정규화 (Denormalization)
JOIN 감소로 조회 성능은 향상되지만, 데이터 중복으로 인해 저장 공간 증가와 무결성 저하라는 트레이드오프가 발생합니다.
기출 "데이터를 삽입할 때 원하지 않는 불필요한 데이터까지 함께 삽입해야만 하는 이상현상"
▼
정답
삽입 이상 (Insertion Anomaly)
정규화가 부족한 테이블에서 발생하는 3가지 이상현상(삽입·삭제·갱신 이상) 중 하나로, 정규화의 필요성을 뒷받침하는 근거입니다.
기출 "인덱스 순서에 맞춰 테이블의 데이터 자체를 물리적으로 정렬하여 저장하며, 한 테이블에 하나만 만들 수 있는 인덱스"
▼
정답
클러스터형 인덱스 (Clustered Index)
별도의 인덱스 구조와 포인터로 데이터를 참조하며 여러 개 생성 가능한 비클러스터형 인덱스와 대비됩니다.
기출 "실제 데이터를 저장하지 않고, SELECT문의 실행 결과를 마치 테이블처럼 사용할 수 있게 하는 가상의 테이블"
▼
정답
뷰 (View)
보안(민감 컬럼 숨김)과 복잡한 쿼리의 단순화에 장점이 있지만, 독립적인 인덱스 생성이 불가능하고 삽입·수정·삭제에 제약이 있습니다.
기출 "INSERT/UPDATE/DELETE 같은 특정 이벤트가 발생하면 별도의 호출 없이 자동으로 실행되는, DB에 저장된 프로그램"
▼
정답
트리거 (Trigger)
CALL/EXEC로 명시적으로 호출해야 하는 프로시저와 달리 DML 이벤트에 종속되어 자동 실행되며, 반환값이 없습니다.
기출 "트랜잭션의 연산이 모두 반영되거나(Commit) 전혀 반영되지 않아야(Rollback) 한다는 ACID 특성"
▼
정답
원자성 (Atomicity)
ACID의 나머지 특성은 일관성(Consistency), 고립성(Isolation), 지속성(Durability)입니다.
03 통합 구현
기출 "여러 이기종 시스템 간의 데이터 및 프로세스를 유기적으로 연동시키기 위한 애플리케이션 통합 기술"
▼
정답
EAI (Enterprise Application Integration)
구축 유형으로 Point-to-Point, Hub&Spoke, Message Bus(ESB), Hybrid가 있습니다.
기출 "모든 연계 시스템이 중앙의 허브를 거쳐서 데이터를 주고받는 EAI 구축 유형 (허브 장애 시 전체 영향)"
▼
정답
Hub & Spoke 방식
중앙집중형 관리가 가능하지만 허브가 단일 장애점(SPOF)이 될 수 있습니다.
기출 "시스템 간을 1:1로 직접 연결하는 가장 단순한 EAI 구축 유형으로, 시스템 수가 늘어날수록 연계 복잡도가 급격히 증가"
▼
정답
Point-to-Point 방식
구현은 단순하지만 시스템이 N개일 때 최대 N(N-1)/2개의 연결이 필요해질 수 있어 확장성이 떨어집니다.
기출 "표준화된 메시지 버스를 통해 시스템을 연계하며, EAI 구축 유형 중 확장성이 가장 우수한 방식"
▼
정답
Message Bus 방식 (ESB 기반)
시스템들이 버스에만 연결하면 되므로 결합도가 낮고 시스템 추가·변경에 유연합니다.
기출 "서비스 중심의 미들웨어로 메시지 라우팅, 데이터 형식·프로토콜 변환을 담당하여 시스템 간 결합도를 낮추는 기술"
▼
정답
ESB (Enterprise Service Bus)
EAI의 발전된 형태로, 서비스 지향 아키텍처(SOA) 구현의 핵심 인프라로 사용됩니다.
기출 "HTTP 메서드(GET/POST/PUT/DELETE)를 이용해 자원을 표현하고 조작하는 경량 아키텍처 스타일"
▼
정답
REST (Representational State Transfer)
주로 JSON을 사용해 가볍고 빠르며, 캐싱이 용이합니다. XML 기반의 무거운 SOAP과 대비됩니다.
심화 "XML 기반으로 메시지를 교환하고 WSDL로 서비스를 명세하며, 보안·트랜잭션 기능은 강력하지만 상대적으로 무거운 프로토콜"
▼
정답
SOAP (Simple Object Access Protocol)
REST보다 형식이 엄격하여 무겁지만, 엔터프라이즈 환경의 보안·신뢰성 요구사항에 강점이 있습니다.
기출 "송신 시스템이 응답을 기다리지 않고 다음 처리를 계속 진행하며, 결과는 콜백이나 큐로 수신하는 연계 방식"
▼
정답
비동기 (Asynchronous) 방식
응답을 기다렸다가 다음 처리를 진행하는 동기(Synchronous) 방식과 대비되며, 처리 지연에 덜 민감합니다.
심화 "EAI/ESB 같은 미들웨어를 경유하여 시스템 간에 연계함으로써, 직접 연결 방식보다 결합도를 낮추는 연계 방식"
▼
정답
간접연계
DB Link나 API 직접 호출로 연결하는 직접연계보다 구축 비용은 높지만 변경에 유연하게 대응할 수 있습니다.
기출 "키-값(key-value) 구조의 경량 데이터 형식으로, XML보다 가볍고 파싱이 빨라 웹·모바일 API 연계에 주로 사용되는 형식"
▼
정답
JSON (JavaScript Object Notation)
태그 기반의 계층적 구조를 갖는 XML은 문서형 데이터나 레거시 연계에 주로 사용됩니다.
04 서버프로그램 구현 & SW공학
기출 "모듈 내부의 모든 구성 요소가 오직 하나의 기능을 수행하기 위해 관련되어 있는, 응집도 중 가장 바람직한 유형"
▼
정답
기능적 응집도 (Functional Cohesion)
응집도는 모듈 '내부' 요소 간의 관련성이며, 높을수록 좋습니다(기능적>순차적>교환적>절차적>시간적>논리적>우연적).
기출 "서로 아무런 관련이 없는 요소들이 우연히 하나의 모듈로 묶인, 응집도 중 가장 나쁜(최저) 유형"
▼
정답
우연적 응집도 (Coincidental Cohesion)
암기법 '기순교절시논우'의 마지막 글자로, 응집도 7종류 중 가장 낮은 품질을 의미합니다.
기출 "다른 모듈의 내부 데이터나 코드를 직접 참조하거나 수정하는, 결합도 중 가장 나쁜(강한) 유형"
▼
정답
내용 결합도 (Content Coupling)
결합도는 모듈 '사이'의 의존 정도이며, 낮을수록 좋습니다(내용>공통>외부>제어>스탬프>자료).
기출 "모듈 간에 꼭 필요한 데이터만 매개변수로 전달하는, 결합도 중 가장 바람직한(가장 낮은) 유형"
▼
정답
자료 결합도 (Data Coupling)
암기법 '내공외제스자'의 마지막 글자로, 자료 구조 전체가 아닌 필요한 값만 전달하여 의존성이 최소화된 이상적인 결합 형태입니다.
기출 "클래스의 인스턴스가 오직 하나만 생성되도록 보장하고, 어디서든 접근 가능한 전역적인 접근점을 제공하는 생성 패턴"
▼
정답
싱글턴 패턴 (Singleton Pattern)
DB 커넥션 풀, 설정 관리 객체처럼 프로그램 전체에서 단 하나만 존재해야 하는 객체에 사용합니다.
기출 "객체 생성 로직을 서브클래스에 위임하여, 어떤 클래스의 인스턴스를 생성할지는 서브클래스가 결정하도록 하는 생성 패턴"
▼
정답
팩토리 메서드 패턴 (Factory Method Pattern)
객체 생성 코드를 클라이언트 코드와 분리해 새로운 제품 클래스가 추가되어도 클라이언트 코드를 수정할 필요가 없게 합니다.
기출 "한 객체의 상태가 변할 때, 그 객체에 의존하는 다른 객체들에게 자동으로 알림이 전달되도록 일대다 의존 관계를 정의하는 행위 패턴"
▼
정답
옵저버 패턴 (Observer Pattern)
주체(Subject)의 상태 변화를 구독 중인 다수의 관찰자(Observer)에게 통보하는 발행-구독 구조로, 이벤트 처리 시스템에 널리 쓰입니다.
기출 "여러 알고리즘을 각각 캡슐화하여 서로 교체 가능하게 만들어, 실행 중에도 알고리즘을 독립적으로 바꿔 쓸 수 있게 하는 행위 패턴"
▼
정답
전략 패턴 (Strategy Pattern)
정렬 알고리즘이나 결제 방식처럼 동일한 목적을 다양한 방법으로 수행해야 할 때, 조건문 대신 알고리즘 객체를 교체하여 사용합니다.
심화 "기존 클래스의 코드를 변경하지 않고, 객체에 새로운 책임(기능)을 동적으로 추가할 수 있게 해주는 구조 패턴"
▼
정답
데코레이터 패턴 (Decorator Pattern)
상속 대신 객체를 여러 겹으로 감싸는(Wrapping) 방식으로 기능을 유연하게 확장합니다.
심화 "서로 호환되지 않는 인터페이스를 가진 클래스들이 함께 동작할 수 있도록 중간에서 변환해주는 구조 패턴"
▼
정답
어댑터 패턴 (Adapter Pattern)
기존 코드를 변경하지 않고 인터페이스가 다른 두 클래스를 연결할 때 사용하며, 220V-110V 변환 플러그에 비유됩니다.
기출 "여러 서브시스템의 복잡한 기능들을 감추고, 단순화된 하나의 통합 인터페이스만을 클라이언트에 제공하는 구조 패턴"
▼
정답
퍼사드 패턴 (Facade Pattern)
서브시스템 내부 구조를 몰라도 창구(퍼사드) 하나만으로 기능을 사용할 수 있게 하여 클라이언트와 서브시스템 간 결합도를 낮춥니다.
05 인터페이스 구현
기출 "인터페이스 설계서에 명시되는 항목으로, 데이터를 보내는 쪽과 받는 쪽의 시스템을 식별하는 정보"

정답

기출 "송신 후 응답을 기다렸다가 그 결과를 받은 뒤에야 다음 처리를 진행하는 인터페이스 연동 방식"
▼
정답
동기 (Synchronous) 방식

기출 "HTTP 응답 코드 중, 요청이 정상적으로 처리되었음을 나타내는 코드대"
▼
정답
2xx (성공)
1xx는 정보성 응답, 3xx는 리다이렉션, 4xx는 클라이언트 오류, 5xx는 서버 오류를 의미합니다.
기출 "HTTP 응답 코드 중, 잘못된 요청이나 인증·권한 문제 등 클라이언트 측의 오류를 나타내는 코드대"
▼
정답
4xx (클라이언트 오류)
대표적으로 401(인증 필요), 403(권한 없음), 404(리소스 없음) 등이 있습니다.
기출 "HTTP 응답 코드 중, 서버 내부의 처리 실패로 발생하는 오류를 나타내는 코드대"
▼
정답
5xx (서버 오류)
대표적으로 500(내부 서버 오류), 503(서비스 이용 불가)이 있습니다.
심화 "분산 환경에서 다양한 언어와 플랫폼의 소프트웨어 테스트를 자동화할 수 있게 지원하는 프레임워크"
▼
정답

기출 "웹 기반으로 팀원들이 협업하며 테스트케이스를 작성·실행할 수 있는 테스트 자동화 도구"
▼
정답
FitNesse

심화 "STAF와 FitNesse를 통합하여 만든 한국형 테스트 자동화 프레임워크"

정답
NTAF (Naver Testing Automation Framework)
국내(네이버)에서 STAF의 분산 실행 환경과 FitNesse의 협업 기능을 결합해 개발한 프레임워크입니다.
기출 "웹 브라우저를 제어하여 웹 애플리케이션 테스트를 자동화하는 대표적인 오픈소스 도구"
▼
정답
Selenium
다양한 브라우저와 언어를 지원하며, 웹 UI 자동화 테스트 도구의 사실상 표준으로 널리 쓰입니다.
기출 "전자서명이나 해시값 비교를 통해 인터페이스로 전송되는 데이터가 위·변조되지 않았음을 검증하는 것"

정답
데이터 무결성 검증
전송 구간 암호화(SSL/TLS), 인증(API Key·토큰), 접근통제와 함께 대표적인 인터페이스 보안 적용 요소입니다.

기출 "전사적으로 공통 적용되는 화면 디자인과 구성 규칙을 정의한, UI 지침보다 상위 개념의 문서"

일관성 (Consistency)

기출 "사용자가 별도의 학습 없이도 쉽게 이해하고 사용할 수 있어야 한다는 UI 설계 원칙"

직관성 (Intuitiveness)
아이콘·메뉴 구조 등이 사용자의 기존 경험과 기대에 부합하도록 설계하는 것을 의미합니다.
심화 "사용자가 정확하고 완전하게 자신의 목표(작업)를 달성할 수 있도록 하는 UI 설계 원칙"

유효성 (Efficiency)
단순히 보기 좋은 것이 아니라 사용자의 업무 목적을 실제로 효율적으로 달성하도록 돕는 원칙입니다.
기출 "시스템이 제공하는 기능들을 트리 형태로 계층적으로 분류하여, 사용자가 원하는 기능을 빠르게 찾을 수 있도록 설계하는 것"
▼
정답
메뉴 구조 (계층적/트리 구조)
깊은 계층 구조는 탐색을 어렵게 하므로, 적절한 깊이와 그룹화가 중요합니다.
기출 "사용자의 행동(클릭, 입력 등)에 대해 시스템이 그 결과를 즉시 알려주는 사용자 편의성 요소"
▼
정답
피드백 (Feedback)
로딩 표시, 완료 메시지 등이 대표적이며, 사용자가 시스템 상태를 오해하지 않도록 돕습니다.
기출 "장애인을 포함한 다양한 특성의 사용자들도 쉽게 이용할 수 있도록 보장하는 사용자 편의성 요소"
▼
정답
접근성 (Accessibility)
스크린 리더 지원, 색상 대비, 키보드 조작 등을 고려하여 설계합니다.
07 애플리케이션 테스트 관리
기출 "입력 값의 범위를 유효/무효 구간으로 나누고 각 구간의 대표값으로 테스트 케이스를 설계하는 기법"
▼
정답
동치 분할(클래스) 검사 (Equivalence Partitioning)
예를 들어 1~100이 유효범위라면 대표값 50 하나만 테스트하는 식입니다. 경계값 근처를 집중 공략하는 경계값 분석과 짝을 이루는 대표적인 블랙박스 기법입니다.
기출 "입력 조건의 유효·무효 경계 부근의 값(최댓값, 최솟값, 경계값±1 등)을 집중적으로 테스트하는 블랙박스 기법"
▼
정답
경계값 분석 (Boundary Value Analysis)
오류는 경계 부분에서 많이 발생한다는 경험에 기반한 기법으로, 1~100이 유효범위면 0, 1, 100, 101을 테스트합니다.
기출 "입력 조건(원인)과 출력 결과(효과) 사이의 논리적 관계를 그래프로 표현하여 테스트케이스를 도출하는 블랙박스 기법"
▼
정답
원인-결과 그래프 검사 (Cause-Effect Graphing)
여러 입력 조건의 조합에 따른 출력을 논리적으로 분석할 때 사용하며, 이후 결정 테이블 형태로 정리되기도 합니다.
기출 "여러 조건과 그에 따른 행동(결과)의 조합을 표 형태로 정리하여 가능한 모든 경우의 수를 테스트하는 블랙박스 기법"
▼
정답
결정 테이블 테스트 (Decision Table Testing)
조건이 많아지면 표의 크기가 커지지만, 조합 누락 없이 체계적으로 테스트케이스를 설계할 수 있습니다.
기출 "시스템이 가질 수 있는 상태와 이벤트에 따른 상태 변화(전이)를 기반으로 설계하는 블랙박스 테스트 기법"
▼
정답
상태 전이 테스트 (State Transition Testing)
로그인→메인→로그아웃처럼 상태가 이벤트에 따라 전이되는 시스템(주문 상태, 로그인 세션 등)을 검증할 때 유용합니다.
기출 "프로그램의 모든 구문(코드 라인)이 최소 한 번은 실행되도록 하는, 화이트박스 커버리지 기법 중 강도가 가장 약한 기준"
▼
정답
구문 커버리지 (Statement Coverage)
커버리지 강도 순서는 구문 < 결정(분기)=조건 < 조건/결정(MC/DC) < 다중조건 < 경로 순으로 강해집니다.
기출 "프로그램 내 모든 분기(if문 등의 참/거짓)가 최소 한 번씩 실행되도록 하는 화이트박스 커버리지 기법"
▼
정답
결정(분기) 커버리지 (Decision/Branch Coverage)
개별 조건식이 아닌 분기문 전체의 참/거짓 결과를 기준으로 하며, 조건 커버리지와는 판단 대상이 다릅니다.
심화 "각 개별 조건식의 참/거짓과 전체 결정(분기)의 참/거짓을 모두 최소 한 번씩 만족시키는, 조건 커버리지보다 강력한 화이트박스 커버리지 기준"
▼
정답
조건/결정 커버리지 (MC/DC, Modified Condition/Decision Coverage)
조건 커버리지와 결정 커버리지를 동시에 만족시키는 기준으로, 항공·의료 소프트웨어처럼 높은 안전성이 요구되는 분야에서 사용됩니다.
기출 "아직 개발되지 않은 상위 모듈을 대신하여, 하향식 통합 테스트에서 하위 모듈을 호출하는 데 사용하는 테스트용 제어 프로그램"
▼
정답
스텁 (Stub)
테스트 대상 모듈이 호출하는, 아직 미완성인 하위 모듈을 임시로 대체하는 가짜 모듈입니다.
기출 "아직 개발되지 않은 상위 모듈을 대체하여, 상향식 통합 테스트에서 완성된 하위 모듈을 호출·구동해주는 테스트용 제어 프로그램"
▼
정답
드라이버 (Driver)
스텁은 하향식에서 하위 모듈을, 드라이버는 상향식에서 상위 모듈을 대체한다는 점에서 방향이 서로 반대입니다.
심화 "테스트 결과가 옳은지 그른지를 판단하는 기준"
▼

교착상태 (Deadlock)
상호배제·점유대기·비선점·순환대기 4가지 필요조건이 모두 충족되어야 발생합니다(암기: 상점비순).
기출 "프로세스가 도착한 순서대로 CPU를 할당하는, 가장 단순한 비선점 스케줄링 기법"
▼
정답
FCFS (First Come First Served, 선입선출)
먼저 도착한 긴 작업 때문에 뒤따르는 작업들이 오래 대기하는 호위 효과(Convoy Effect)가 단점입니다.
기출 "정해진 시간 할당량(Time Quantum)만큼씩 프로세스에 CPU를 순환 배분하는 선점형 스케줄링 기법"
▼
정답
라운드 로빈 (Round Robin)
시분할 시스템에 적합하며, 시간 할당량이 너무 크면 FCFS와 유사해지고 너무 작으면 문맥교환 오버헤드가 커집니다.
기출 "메모리를 고정된 크기의 블록으로 나누어 관리하며, 외부 단편화는 없지만 내부 단편화가 발생할 수 있는 메모리 관리 기법"
▼
정답
페이징 (Paging)
고정 크기 페이지 단위로 나누기 때문에 마지막 페이지에서 남는 공간(내부 단편화)이 생길 수 있습니다.
심화 "메모리를 프로그램의 논리적 의미 단위로 나누어 관리하며, 내부 단편화는 없지만 외부 단편화가 발생할 수 있는 메모리 관리 기법"
▼

가변 크기의 논리적 단위(세그먼트)로 나누기 때문에 페이징과 반대되는 단편화 특성을 가집니다.
