01 상속 심화
오버로딩은 매개변수 목록이 다른 별도 메서드(컴파일타임 정적 바인딩), 오버라이딩은 상속 관계에서 동일 시그니처 재정의(런타임 동적 바인딩). 필드 접근은 항상 선언(참조) 타입 기준으로 정적 바인딩된다.
기출변형 다음 Java 코드의 출력 결과를 쓰시오. (오버로딩 해석 — 위해닝 vs 박싱)
▼ 
public class Main {
    static void call(int x) { System.out.print("int "); }
    static void call(long x) { System.out.print("long "); }
    static void call(Integer x) { System.out.print("Integer "); }
    public static void main(String[] args) {
        call(5);
        byte b = 3;
        call(b);
    }
}
단계호출오버로드 해석출력
①call(5)int 리터럴 → call(int) 정확히 일치int
②byte b = 3b는 byte 타입
③call(b)박싱 없는 1단계 탐색: byte→int, byte→long 모두 위해닝 가능 → 더 구체적인 call(int) 선택 (call(Integer)는 박싱 필요해 후순위)int
정답
int int
오버로딩 해석은 1단계(박싱 없는 위해닝만 허용)에서 적용 가능한 메서드를 먼저 찾는다. byte는 int·long 모두로 위해닝되지만 더 구체적인 call(int)가 선택되고, 박싱이 필요한 call(Integer)는 1단계에서 아예 후보가 되지 않는다.
기출변형 다음 Java 코드의 출력 결과를 쓰시오. (생성자에서 오버라이드 메서드 호출)
▼ 
class Parent {
    Parent() {
        init();
    }
    void init() { System.out.print("Parent-init "); }
}
class Child extends Parent {
    int value = 10;
    @Override
    void init() { System.out.print("Child-init:" + value + " "); }
}
public class Main {
    public static void main(String[] args) {
        new Child();
    }
}
단계실행 위치value출력
①new Child() → 암묵적 super() 호출 → Parent() 진입-
②Parent() 내부에서 init() 호출 → 동적 바인딩 → Child.init() 실행0 (필드 초기화 전 기본값)Child-init:0
③Parent() 리턴 → Child의 필드 초기화 실행10
④Child() 본문 (없음, 기본 생성자)10
정답
Child-init:0
super() 실행 중 호출된 init()은 동적 바인딩으로 Child.init()이 실행되지만, 이 시점은 Child의 필드 초기화(value=10)가 아직 수행되기 전이므로 value는 기본값 0이다. 생성자에서 오버라이드 가능한 메서드를 호출하면 안 되는 대표적 이유.
심화 다음 Java 코드의 출력 결과를 쓰시오. (3단계 상속 + super 체인)
▼ 
class A {
    void show() { System.out.print("A"); }
}
class B extends A {
    @Override
    void show() { super.show(); System.out.print("B"); }
}
class C extends B {
    @Override
    void show() { super.show(); System.out.print("C"); }
}
public class Main {
    public static void main(String[] args) {
        A obj = new C();
        obj.show();
    }
}
단계호출출력
①obj.show() → 동적 바인딩 → C.show() 진입
②C.show(): super.show() → B.show() 진입
③B.show(): super.show() → A.show() 진입
④A.show() 실행A
⑤B.show() 복귀 후 나머지 실행B
⑥C.show() 복귀 후 나머지 실행C
정답
ABC
super.show()는 항상 "한 단계 위" 클래스의 메서드를 호출하므로 A→B→C 순으로 실행되어 최종 출력은 ABC.
기출변형 다음 Java 코드의 출력 결과를 쓰시오. (오버로딩 vs 오버라이딩 구분)
▼ 
class Parent {
    void greet(Object o) { System.out.print("Parent-Object "); }
}
class Child extends Parent {
    void greet(String s) { System.out.print("Child-String "); } // 오버로딩(시그니처 다름), 오버라이딩 아님

①p의 선언 타입 = Parent → 컴파일러가 Parent 안에서 greet 후보 탐색greet(Object)만 보임
②Child.greet(String)은 시그니처가 달라 오버라이딩 아닌 별개 오버로드 → Parent 참조로는 안 보임-
③런타임에도 Child가 greet(Object)를 재정의하지 않았으므로 Parent.greet(Object) 그대로 실행Parent-Object
정답
Parent-Object
Child.greet(String)은 매개변수 타입이 달라 오버라이딩이 아니라 새로운 오버로드다. p의 선언 타입이 Parent이므로 컴파일 시점에 greet(Object)로 바인딩되고, Child가 greet(Object)를 재정의하지 않았으므로 Parent의 구현이 그대로 실행된다.
심화 다음 Java 코드의 출력 결과를 쓰시오. (공변 반환타입 + 필드 정적 바인딩)
▼ 
class A {
    int x = 1;
    A get() { System.out.print("A.get "); return this; }
}
class B extends A {
    int x = 2;
    @Override
    B get() { System.out.print("B.get "); return this; } // 공변 반환타입
}
public class Main {
    public static void main(String[] args) {
        A a = new B();
        System.out.print(a.get().x);
    }
}
단계판단결과
①a.get() → 메서드는 동적 바인딩 → 실제 타입 B의 get() 실행B.get
②a.get() 표현식의 컴파일타임 타입은 a의 선언 타입 A 기준 A.get()의 반환형인 A로 고정됨 (공변 반환은 실제 객체엔 영향, 식의 정적 타입엔 영향 없음)정적 타입 = A
③.x 필드 접근은 정적 바인딩 → 정적 타입 A의 x 필드 접근1
정답
B.get 1
get() 호출은 동적 바인딩으로 B.get()이 실행되어 "B.get "이 출력되지만, a.get().x의 .x 필드 접근은 a.get() 식의 정적(컴파일타임) 타입인 A를 기준으로 정적 바인딩되므로 A의 x(=1)가 접근된다. 메서드는 동적, 필드는 정적이라는 원칙이 공변 반환타입과 결합된 심화 함정.
02 인터페이스 & 추상클래스
인터페이스 필드는 암묵적으로 public static final. default 메서드가 충돌하면 반드시 재정의해야 하며 인터페이스명.super.메서드()로 특정 구현을 호출한다. 클래스(추상클래스 포함)의 메서드는 인터페이스 default보다 항상 우선한다.
기출변형 다음 Java 코드의 출력 결과를 쓰시오. (다이아몬드 default 충돌 해결)
▼ 
interface A {
    default String hi() { return "A-hi"; }
}
interface B {
    default String hi() { return "B-hi"; }
}
class C implements A, B {
    @Override
    public String hi() {
        return A.super.hi() + B.super.hi();
    }
}
public class Main {
    public static void main(String[] args) {
        System.out.println(new C().hi());
    }
}
단계동작값
①C가 A, B를 동시 구현 → 두 default hi()가 충돌 → 반드시 재정의 필요(안 하면 컴파일 에러)-
②A.super.hi() 호출"A-hi"
③B.super.hi() 호출"B-hi"
④문자열 연결"A-hiB-hi"
정답
A-hiB-hi
두 인터페이스가 같은 시그니처의 default 메서드를 가지면 구현 클래스가 반드시 재정의해야 한다. 이때 인터페이스명.super.메서드()로 특정 인터페이스의 default 구현을 명시적으로 호출할 수 있다.
기출변형 다음 Java 코드의 출력 결과를 쓰시오. (추상클래스 템플릿 메서드)
▼ 
abstract class Shape {
    String name;
    Shape(String name) { this.name = name; }
    abstract int area();
    void report() { System.out.print(name + "=" + area() + " "); }
}
class Square extends Shape {
    int side;
    Square(int side) { super("Square"); this.side = side; }
    @Override
    int area() { return side * side; }
}
public class Main {
    public static void main(String[] args) {
        Shape s = new Square(4);
        s.report();
    }
}
단계동작값
①new Square(4) → super("Square") → name="Square", side=4name=Square, side=4
②s.report() 실행 → 내부에서 area() 동적 바인딩 → Square.area()area() = 4*4 = 16
③"Square=16 " 출력Square=16
정답
Square=16
추상클래스의 일반 메서드(report)는 그대로 상속되어 실행되지만, 내부에서 호출하는 추상 메서드(area)는 실제 객체(Square)의 구현으로 동적 바인딩된다.
심화 다음 Java 코드의 출력 결과를 쓰시오. (인터페이스 상수 + default)
▼ 
interface Flyable {
    int MAX_SPEED = 100; // 암묵적 public static final

public class Main {

단계동작출력
①f.fly() → Bird.fly() 실행flying
②f.info() → 재정의 없어 Flyable의 default 실행, MAX_SPEED=100MAX=100
③Flyable.MAX_SPEED 직접 접근 (static 상수)100
정답
flying MAX=100 100
인터페이스에 선언한 필드는 자동으로 public static final 상수가 되어 인스턴스 없이 인터페이스명으로 접근 가능하다. Bird가 info()를 재정의하지 않았으므로 default 구현이 사용된다.
심화 다음 Java 코드의 출력 결과를 쓰시오. (함수형 인터페이스 + default 템플릿)
▼ 
interface Validator {
    boolean isValid(int n);
    default void check(int n) {
        if (isValid(n)) System.out.print(n + ":OK ");
        else System.out.print(n + ":FAIL ");
    }
}
public class Main {
    public static void main(String[] args) {
        Validator positive = n -> n > 0;
        positive.check(5);
        positive.check(-3);
    }
}
단계동작결과
①positive = 람다 (n -> n > 0) → Validator.isValid 구현-
②check(5) → isValid(5) = true5:OK
③check(-3) → isValid(-3) = false-3:FAIL
정답
5:OK -3:FAIL
추상 메서드가 1개인 Validator는 함수형 인터페이스라 람다로 구현 가능하다. default 메서드 check()는 람다가 구현한 isValid()를 내부에서 호출하는 템플릿 메서드 패턴.
심화 다음 Java 코드의 출력 결과를 쓰시오. (클래스 우선 규칙)
▼ 
interface Greeter {
    default String greet() { return "Interface-Hi"; }
}
abstract class Base implements Greeter {
    @Override
    public String greet() { return "Base-Hi"; }
}
class Impl extends Base {
    // greet() 재정의 없음
}
public class Main {
    public static void main(String[] args) {
        Greeter g = new Impl();
        System.out.println(g.greet());
    }
}
단계판단결과
①Impl은 greet() 미재정의 → Base의 greet() 상속-
②"클래스 메서드가 인터페이스 default보다 항상 우선" 규칙 적용 → Base.greet() 사용, Greeter의 default는 완전히 가려짐-
③g.greet() 실행Base-Hi
정답
Base-Hi
추상클래스라도 클래스 계층에서 명시적으로 정의된 메서드는 인터페이스의 default 메서드보다 항상 우선한다("class wins" 규칙). Impl은 Base.greet()를 그대로 물려받으므로 Base-Hi가 출력된다.
03 예외처리 심화
finally 안에 return이 있으면 try/catch의 예외·반환값을 모두 덮어써 버린다(예외 삼켜짐 주의). try-with-resources는 선언 순서대로 열리고 역순으로 close()된다.
기출변형 다음 Java 코드의 출력 결과를 쓰시오. (멀티 catch)
▼ 
public class Main {
    static void test(int n) {
        try {
            if (n == 0) throw new ArithmeticException("A");
            else throw new NullPointerException("N");
        } catch (ArithmeticException | NullPointerException e) {
            System.out.print(e.getMessage() + " ");
        } finally {
            System.out.print("F ");
        }
    }
    public static void main(String[] args) {
        test(0);
        test(1);
    }
}
단계호출발생 예외출력
①test(0)ArithmeticException("A")A F
②test(1)NullPointerException("N")N F
정답
A F N F
멀티 catch(A | B)는 두 예외 타입 중 하나라도 발생하면 한 블록에서 함께 처리한다. finally는 두 호출 모두에서 항상 실행된다.
기출변형 다음 Java 코드의 출력 결과를 쓰시오. (예외 체이닝)
▼ 
public class Main {
    static void inner() {
        throw new RuntimeException("inner-fail");
    }
    static void outer() {
        try {
            inner();
        } catch (RuntimeException e) {
            throw new IllegalStateException("outer-fail", e);
        }
    }
    public static void main(String[] args) {
        try {
            outer();
        } catch (IllegalStateException e) {
            System.out.print(e.getMessage() + "/" + e.getCause().getMessage());
        }
    }
}
단계동작상태
①inner() 실행 → RuntimeException("inner-fail") 발생-
②outer()의 catch에서 잡음 → 새 IllegalStateException("outer-fail", e)를 원인(cause)와 함께 던짐cause=inner-fail
③main의 catch에서 잡음 → getMessage()="outer-fail", getCause().getMessage()="inner-fail"outer-fail/inner-fail
정답
outer-fail/inner-fail
예외 체이닝은 원래 예외(cause)를 감싸 새 예외로 던져 문맥 정보를 보존하는 기법이다. getCause()로 원본 예외를 추적할 수 있다.
심화 다음 Java 코드의 출력 결과를 쓰시오. (finally와 return 값의 시점)
▼ 
public class Main {
    static int test() {
        int x = 1;
        try {
            return x;
        } finally {
            x = 99;
            System.out.print("finally ");
        }
    }
    public static void main(String[] args) {
        System.out.print(test());
    }
}
단계동작x반환값(확정)
①x = 11-
②return x → 반환값 1을 미리 복사·확정11
③finally 실행: x=99, "finally " 출력991 (변경 안됨)
④메서드 종료, 확정된 값 1 반환-1
⑤System.out.print(1)-1
정답
finally 1
기본형 반환값은 return 문 실행 시점에 이미 복사·확정되므로, 이후 finally에서 지역변수 x를 바꿔도 반환값에는 영향이 없다. 출력 순서상 "finally "가 먼저, 반환된 1이 나중에 출력된다.
심화 다음 Java 코드의 출력 결과를 쓰시오. (finally의 return이 예외를 삼킴)
▼

②finally 블록 실행 중 return 42 만남finally가 자체 return을 가짐
③finally의 return이 전파 중이던 예외를 완전히 대체 → 예외는 사라지고 42가 정상 반환예외 소멸

finally 블록 안에 return 문이 있으면 try/catch에서 전파되던 예외를 무조건 덮어써서 삼켜버린다. 컴파일은 되지만 예외를 숨기는 매우 위험한 패턴으로, 실무에서는 절대 사용하면 안 된다.
심화 다음 Java 코드의 출력 결과를 쓰시오. (try-with-resources 다중 자원)
▼ 
class Res implements AutoCloseable {
    String name;
    Res(String name) { this.name = name; System.out.print("open" + name + " "); }
    @Override
    public void close() { System.out.print("close" + name + " "); }
}

        }
    }
}

PECS 원칙: 읽기만 하면 ? extends T(Producer), 쓰기만 하면 ? super T(Consumer). 제네릭은 컴파일 후 타입 소거(Type Erasure)되어 런타임에는 모두 같은 raw 타입이 된다.

        Box<Integer> box = new Box<>();
        box.set(10);

    }
}
단계동작value
①Box<Integer> box = new Box<>()null
②box.set(10) → T=Integer로 오토박싱 저장10
③box.get() + 5 → 10 + 515
정답

06 Stream API
Stream 연산은 중간 연산(filter/map/sorted 등, 지연 실행)과 최종 연산(collect/reduce/sum 등, 실제 실행 트리거)으로 나뉜다. 스트림은 1회용이라 한 번 소비하면 재사용할 수 없다.
기출변형 다음 Java 코드의 출력 결과를 쓰시오. (filter + map + collect)

import java.util.*;
import java.util.stream.*;
public class Main {

import java.util.*;
import java.util.stream.*;
public class Main {

10 스레드 기초
start()는 새 스레드에서 run()을 실행하지만 run()을 직접 호출하면 새 스레드 없이 현재 스레드에서 그냥 메서드처럼 실행된다. synchronized는 같은 락 객체를 사용할 때만 상호배제 효과가 있다.
기출변형 다음 Java 코드의 출력 결과를 쓰시오. (run() 직접 호출 vs start())

public class Main {
    public static void main(String[] args) {
        Runnable r = () -> System.out.print("Run ");
        r.run();                    // 그냥 메서드 호출 (새 스레드 아님)
        Thread t = new Thread(r);
        System.out.print("Before-start ");
        t.run();                    // start()가 아닌 run() 직접 호출 → 메인 스레드에서 실행
        System.out.print("End");
    }
}

public class Main {

            threads[i] = new Thread(() -> {

}

③5+1=6 쓰기-6

counter++는 "읽기→증가→쓰기" 세 단계로 이루어져 원자적 연산이 아니다. synchronized가 없으면 두 스레드가 동시에 같은 값을 읽고 각자 +1한 뒤 써서 한쪽의 증가가 유실되는 race condition(경쟁 상태)이 발생할 수 있다. 따라서 최종 counter는 5000이 아니라 그보다 작은 값이 될 가능성이 있으며 실행할 때마다 결과가 달라질 수 있다.
심화 다음 Java 코드의 마지막 출력 결과를 쓰고 그 이유를 설명하시오. (서로 다른 락 객체 — 동기화 무효)
▼ 
public class Main {
    static int counter = 0;
    static final Object lockA = new Object();
    static final Object lockB = new Object();
    static void incA() {
        synchronized (lockA) { counter++; }
    }
    static void incB() {
        synchronized (lockB) { counter++; } // 다른 락! 동기화 효과 없음
    }
    public static void main(String[] args) throws InterruptedException {
        Thread t1 = new Thread(() -> { for (int i = 0; i < 1000; i++) incA(); });
        Thread t2 = new Thread(() -> { for (int i = 0; i < 1000; i++) incB(); });
        t1.start(); t2.start();
        t1.join(); t2.join();
        System.out.println(counter <= 2000);
    }
}
단계분석결론
①incA()는 lockA, incB()는 lockB로 각각 다른 모니터에 synchronized → t1과 t2는 서로를 배제하지 못함(각자 자기 스레드끼리만 순차적이지만 스레드는 1개씩이라 의미 없음)t1·t2 사이 race condition 여전히 가능
②counter++가 중간에 끼어들 수 있어 counter의 정확한 값(2000)은 보장되지 않음counter ≤ 2000은 항상 성립 (총 시도 횟수가 2000번이므로 유실은 있어도 초과할 수 없음)
③println(counter <= 2000) — counter가 정확히 얼마이든 이 비교식 자체는 항상 참true
정답
true
synchronized는 "같은 락 객체"를 두고 경쟁할 때만 상호배제를 보장한다. incA와 incB가 서로 다른 락(lockA, lockB)을 사용하므로 t1과 t2 사이에는 사실상 동기화 효과가 없어 counter++의 race condition이 발생할 수 있고, 그 결과 counter는 2000이 되지 않을 수도 있다. 다만 총 증가 시도 횟수 자체가 2000번을 넘지 않으므로 counter는 절대 2000을 초과할 수 없어 counter <= 2000은 실행할 때마다 항상 true로 출력된다.
