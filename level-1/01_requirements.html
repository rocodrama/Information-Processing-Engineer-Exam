2 핵심 개념

현행 시스템 파악 순서

순서파악 항목내용
1구성 현황업무 단위, 하위 시스템 구성
2기능 현황각 업무가 처리하는 기능 목록
3인터페이스 현황타 시스템과의 데이터 교환 방식
4아키텍처 현황전체 구조, 계층 구조
5소프트웨어 현황OS, DBMS, 미들웨어 버전
6하드웨어 현황서버 사양, 수량, 위치
7네트워크 현황네트워크 구성도, 회선 종류/용량

미들웨어 (Middleware) 종류

종류설명예시
WAS
Web Application Server		 동적 웹 콘텐츠 처리, 비즈니스 로직 수행 Tomcat, WebLogic, WebSphere, JBoss
MOM
Message Oriented Middleware		 메시지 큐 기반 비동기 통신 RabbitMQ, ActiveMQ, Kafka
TP-Monitor 트랜잭션 처리 모니터링, 분산 트랜잭션 관리 Tuxedo, CICS
ORB
Object Request Broker		 분산 환경에서 객체 간 통신 지원 CORBA
RPC
Remote Procedure Call		 원격 프로시저를 로컬처럼 호출 gRPC
DB 미들웨어 애플리케이션과 DBMS 사이 연결 ODBC, JDBC

기능적 vs 비기능적 요구사항 (자주 출제!)

구분정의예시
기능적 요구사항
Functional		 시스템이 무엇을 해야 하는가
기능, 동작, 처리에 관한 요구사항		 • 로그인/로그아웃 기능
• 주문 처리 기능
• 보고서 출력 기능
• 회원 가입 기능
비기능적 요구사항
Non-Functional		 시스템이 얼마나 잘 해야 하는가
품질, 제약, 성능에 관한 요구사항		 • 응답시간 3초 이내
• 가용성 99.9% 이상
• 보안 암호화 적용
• 동시 사용자 1000명 처리
비기능적 요구사항 세부: 성능 · 보안 · 가용성 · 유지보수성 · 이식성 · 확장성

SRS (소프트웨어 요구사항 명세서) 특성

특성설명
정확성요구사항이 정확하게 기술되어야 함
완전성모든 요구사항이 빠짐없이 포함
일관성요구사항 간 상충·모순 없어야 함
명확성누구나 같은 의미로 해석할 수 있어야 함
검증 가능성테스트를 통해 충족 여부 확인 가능
추적 가능성출처와 이후 설계·구현에 추적 가능
변경 용이성요구사항 변경이 용이하게 구조화

UML 다이어그램 분류

구조 다이어그램
Structural — 정적 구조		 행위 다이어그램
Behavioral — 동적 행위
• 클래스 — 클래스 간 관계
• 객체 — 클래스 인스턴스
• 컴포넌트 — SW 컴포넌트
• 배치 — HW 배치
• 패키지 — 패키지 간 관계 		• 유스케이스 — 기능 요구사항
• 시퀀스 — 메시지 시간 순서
• 활동 — 처리 흐름
• 상태 — 상태 변화
• 협력/통신 — 객체 간 관계

주요 UML 다이어그램 용도

다이어그램분류용도
유스케이스 행위 액터(Actor)와 시스템의 상호작용, 기능 요구사항 표현
클래스 구조 클래스 속성·메서드·관계 (집합, 상속, 의존, 연관)
시퀀스 행위 객체 간 메시지 교환 순서, 시간 흐름에 따른 상호작용
활동 행위 업무 처리 흐름, 조건 분기, 병렬 처리 표현
상태 행위 객체의 상태와 상태 전이 조건
컴포넌트 구조 소프트웨어 컴포넌트 간 의존 관계
배치 구조 하드웨어 노드에 SW 컴포넌트 배치 현황

클래스 다이어그램 관계 종류

관계표기설명
연관 (Association)────클래스 간 일반적인 관계
집합 (Aggregation)◇────전체-부분 관계 (부분이 독립 존재 가능)
복합 (Composition)◆────강한 전체-부분 (전체 소멸 시 부분도 소멸)
일반화 (Generalization)△────상속 관계 (is-a)
의존 (Dependency)------>한 클래스가 다른 클래스를 일시적으로 사용
실체화 (Realization)------△인터페이스 구현 관계

요구사항 검증 기법 (자주 출제!)

기법설명특징
동료 검토
Peer Review		 작성자가 명세서를 설명하고 다른 팀원들이 결함을 발견 비형식적, 소규모
워크쓰루
Walkthrough		 개발자가 주도하여 소수의 동료들과 함께 검토 비형식적, 개발자 주도
인스펙션
Inspection		 진행자(Moderator)가 주도, 체크리스트 기반 공식 검토 가장 공식적, 역할 분리 (작성자≠검토자)
프로토타이핑 초기 버전을 빠르게 만들어 사용자 검증 사용자 피드백 빠름, 비용 발생
테스트 설계 요구사항 기반으로 테스트 케이스를 미리 작성하여 검증 테스트 가능성 확인
공식성 순서: 동료검토 < 워크쓰루 < 인스펙션 (인스펙션이 가장 공식적)

요구사항 도출 기법

기법설명
인터뷰이해관계자와 직접 대화하여 요구사항 수집
설문조사다수를 대상으로 구조화된 질문
브레인스토밍자유로운 아이디어 제안, 비판 금지
프로토타이핑시제품 개발로 요구사항 구체화
유스케이스사용자-시스템 상호작용 시나리오 기반
관찰사용자 업무 현장 직접 관찰
워크샵관계자 집합하여 집중적으로 요구사항 도출
