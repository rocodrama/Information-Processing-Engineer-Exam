2 핵심 개념

보안의 3대 요소 (CIA, 자주 출제!)

요소영문의미
기밀성Confidentiality인가된 사용자만 정보에 접근할 수 있도록 함
무결성Integrity정보가 비인가자에 의해 변경되지 않도록 보장
가용성Availability인가된 사용자가 필요할 때 정보에 접근 가능하도록 보장
추가 요소: 인증(Authentication), 부인방지(Non-repudiation), 책임추적성(Accountability)

보안 설계 원칙 (Saltzer & Schroeder 8원칙)

원칙설명
최소 권한필요한 최소한의 권한만 부여
Fail-safe Default권한이 없으면 기본적으로 접근을 차단
완전한 조정모든 접근 시도를 매번 검사
개방형 설계보안 메커니즘은 비밀이 아닌 공개된 설계로 안전성을 확보 (보안=알고리즘 공개해도 키만 안전하면 됨)
권한 분리하나의 권한으로 전체를 통제하지 못하게 분리
최소 공통 메커니즘공유 자원/메커니즘 최소화로 정보 누설 경로 차단
심리적 수용성사용자가 사용하기 편리해야 보안 절차를 따름
경제적 효율성(Economy of Mechanism)보안 메커니즘은 단순하고 작게 설계

화이트리스트 vs 블랙리스트

구분방식안전성
화이트리스트허용할 값만 정의하고 나머지는 모두 차단더 안전 (권장)
블랙리스트금지할 값만 정의하고 나머지는 모두 허용우회 가능성 있음

주요 입력값 검증 취약점

취약점설명대응
SQL Injection입력값에 SQL 구문을 삽입해 DB 비정상 조작Prepared Statement(바인드 변수)
XSS악성 스크립트를 웹페이지에 삽입·실행입력 검증 + 출력 HTML 인코딩
Command Injection입력값으로 시스템 명령어를 삽입·실행외부 명령 호출 최소화, 입력값 검증
경로조작/디렉터리 리스팅경로 문자열 조작으로 임의 파일 접근경로 정규화, 디렉터리 검색기능 차단
버퍼 오버플로우할당된 메모리보다 큰 데이터로 메모리 영역 침범입력 길이 검증, strncpy 등 안전한 함수 사용

인증의 3요소

유형영문예시
지식 기반Something you know비밀번호, 보안질문
소유 기반Something you haveOTP 토큰, 스마트카드
생체 기반Something you are지문, 홍채, 정맥

접근통제 모델

모델전체명특징
MACMandatory Access Control시스템(보안관리자)이 강제로 권한 결정, 변경 어려움
DACDiscretionary Access Control데이터 소유자가 권한을 직접 결정
RBACRole-Based Access Control사전 정의된 역할(Role)에 따라 권한 부여

암호화 알고리즘 분류

분류특징대표 알고리즘
대칭키(비밀키)암·복호화에 같은 키 사용, 속도 빠름, 키 분배 문제DES, AES, SEED
비대칭키(공개키)공개키/개인키 쌍 사용, 키 분배 용이, 속도 느림RSA, ECC
해시(Hash)단방향, 원문 복원 불가, 무결성 검증·비밀번호 저장SHA-256, MD5
비밀번호는 해시(+Salt)로 저장 — 복호화 가능한 양방향 암호화는 부적절!
Salt: 해시 저장 시 추가하는 임의값 → 레인보우테이블 공격 방지

시큐어코딩 가이드 7대 분류 (KISA, 매회 출제!)

분류대표 취약점
입력데이터 검증 및 표현SQL Injection, XSS, 경로조작, Command Injection
보안기능부적절한 인가, 취약한 암호화, 하드코딩된 비밀번호
시간 및 상태경쟁조건(Race Condition), 세션 고정
에러처리오류 메시지를 통한 정보 노출
코드오류널 포인터 역참조, 자원 누수, 부적절한 자원 해제
캡슐화중요 정보의 부적절한 캡슐화, 디버그 코드 잔존
API 오용위험한 API 사용, 신뢰할 수 없는 데이터의 API 사용

정적 분석 vs 동적 분석

구분방식특징
정적 분석프로그램을 실행하지 않고 소스코드 자체를 분석코드 레벨 취약점 조기 발견 (예: SonarQube)
동적 분석프로그램을 실제 실행하며 분석런타임 동작 기반 취약점 탐지 (예: Fuzzing)

결함 관리 프로세스

결함 발견 → 결함 등록 → 결함 분석(원인/심각도) → 결함 수정 → 재테스트 → 결함 종료
