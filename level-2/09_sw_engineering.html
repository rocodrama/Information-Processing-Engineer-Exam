02 응집도 (Cohesion) — 높을수록 좋음
모듈 내부 요소들이 얼마나 밀접하게 관련되어 있는지. 높은 응집도 = 좋은 설계. 암기: 기순교절시논우

응집도 수준 (낮음 → 높음)

우연적 (Coincidental)
가장 낮음 — 관련 없는 기능 모음
논리적 (Logical)
유사 기능 묶음 (if로 선택)
시간적 (Temporal)
같은 시간대 실행 (초기화 루틴)
절차적 (Procedural)
순서에 따라 실행 (파일 열기→읽기→닫기)
교환적 (Communicational)
같은 데이터 사용 또는 생성
순차적 (Sequential)
출력이 다음 입력 (파이프라인)
기능적 (Functional)
가장 높음 — 단 하나의 기능만 수행
✅ 암기: 우논시절교순기 (낮→높) or 기순교절시논우 (높→낮)
03 결합도 (Coupling) — 낮을수록 좋음
모듈 간 상호 의존 정도. 낮은 결합도 = 좋은 설계. 암기: 자스제외공내

결합도 수준 (낮음 → 높음)

자료 (Data)
가장 낮음 — 단순 값(스칼라) 전달
스탬프 (Stamp)
자료구조(구조체·클래스) 전달
제어 (Control)
제어 신호(플래그) 전달로 흐름 제어
외부 (External)
외부 환경(파일형식·통신프로토콜) 공유
공통 (Common)
전역 데이터를 여러 모듈이 공유
내용 (Content)
가장 높음 — 내부 코드·변수 직접 참조
✅ 암기: 자스제외공내 (낮→높)
자료: int 전달 / 스탬프: struct 전달 / 제어: flag 전달 / 외부: 파일 공유 / 공통: global 변수 / 내용: 내부직접접근

"어떤 클래스의 인스턴스가 오직 하나임을 보장하고, 이에 대한 전역 접근점을 제공한다."

정답
Singleton (싱글톤)
생성 패턴에 속함. 클래스 자신이 유일한 인스턴스를 관리.
구현: private 생성자 + static getInstance() 메서드
활용: DB 연결, 로그 매니저, 설정 파일 관리자

"객체의 상태가 변할 때 그 객체에 의존하는 다른 객체들에게 자동으로 알리고 갱신되도록 한다. 발행-구독 패턴이라고도 한다."

Observer (옵저버)
행위 패턴. Subject(발행자) 1 : Observer(구독자) N 구조.
Subject 상태 변화 시 모든 Observer에게 자동으로 notify().
활용: MVC 모델→뷰 갱신, 이벤트 리스너, 알림 시스템

정답
응집도: 높게 / 결합도: 낮게
우연 → 논리 → 시간 → 절차 → 교환 → 순차 → 기능
응집도 높게: 한 모듈이 하나의 기능만 수행 (기능적 응집도 지향)
결합도 낮게: 모듈 간 의존성 최소화 (자료 결합도 지향)

응집도 낮→높: 우논시절교순기
우연적 < 논리적 < 시간적 < 절차적 < 교환적 < 순차적 < 기능적

"알고리즘의 골격을 상위 클래스에서 정의하고, 알고리즘의 일부 단계는 하위 클래스에서 구현한다."

정답
Template Method (템플릿 메서드)
행위 패턴. 상위 클래스에 알고리즘 흐름(골격) 정의, 하위 클래스에서 세부 구현.
Hollywood Principle: "Don't call us, we'll call you" (부모가 자식 메서드 호출)
활용: 게임 초기화-플레이-종료 흐름, 요리 레시피(재료준비-조리-완성)
기출 유사  화이트박스 테스트의 결정(분기) 커버리지를 설명하고, 구문 커버리지와의 차이를 서술하시오.
▼
정답
결정 커버리지: 모든 분기(if/switch)의 True·False 각 1회 이상 실행
구문(문장) 커버리지: 모든 코드 라인이 최소 1번 실행되면 100%
→ if(A) { X } 에서 A=True만 테스트해도 구문 커버리지 달성 가능

결정(분기) 커버리지: if의 True 분기도, False 분기도 각각 최소 1번
→ A=True 케이스 + A=False 케이스 모두 필요

결정 커버리지 100% ⊃ 구문 커버리지 100% (결정이 더 강한 조건)
기출 유사  나선형 모델의 4단계를 순서대로 쓰고, 다른 모델과 비교한 핵심 특징을 서술하시오.
▼
정답
계획 수립 → 위험 분석 → 개발 및 검증 → 고객 평가 (반복)
나선형 모델 4단계:
① 계획 수립: 목표·대안·제약 정의
② 위험 분석: 위험 요소 식별·해결 방안 검토 (핵심!)
③ 개발 및 검증: 프로토타입 개발·테스트
④ 고객 평가: 사용자 피드백 → 다음 반복으로

핵심 특징: 위험 관리(Risk Management) 강조. 폭포수+프로토타입 결합.
대규모·고위험·장기 프로젝트에 적합. Boehm이 제안.
